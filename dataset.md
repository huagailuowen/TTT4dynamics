# Dataset Generation Notes

## 1. Friction-Randomized Push Dataset

### LIBERO Push-Box Friction Dataset

This dataset is a physical-property adaptation benchmark. The robot pushes a box on a table with different MuJoCo friction coefficients. The intended behavior is:

1. close the gripper;
2. move to the side of the box;
3. make one short, fast, smooth push;
4. retreat;
5. let the box slide freely into the green target region.

The push must not be generated by repeatedly moving the end effector backward and forward at high frequency. That behavior can reach the target, but it is hard for a policy to learn and does not represent a clean physical impulse. The current generator therefore uses a one-shot impulse push.

### Files

- `ttt4dynamics/push_box_libero.py`
  - LIBERO / MuJoCo environment wrapper.
  - Applies friction to the box and table collision geoms.
  - Implements scripted approach, descend, push, retreat, and settle phases.
- `scripts/sweep_libero_push_box_impulse.py`
  - Sweeps impulse-push parameters for one friction value.
  - Reports final distance, minimum distance, end-effector step size, and backward-motion checks.
  - Stops when it finds a successful case with no backward push oscillation.
- `scripts/generate_libero_push_box_adaptation_dataset.py`
  - Generates BDDL files and calibrated case configs for source/adaptation groups.
- `scripts/render_libero_push_box_friction_cases.py`
  - Renders cases to videos and writes `metadata.json`.

### Control Design

The old approach used large global `controller_output_scale` values. That is unsafe for this dataset: it changes the controller scale for all phases, including retreat and settle, and can make the IK / OSC solver jump between joint configurations.

The fixed approach separates high-speed pushing from normal arm motion:

- `controller_output_scale = 1.0`
- `enable_controller_output_scaling = false`
- normal approach / descend / retreat use the standard controller scale;
- only the push phase uses `pusher_push_controller_scale`;
- push uses `pusher_push_mode = "impulse"`;
- the impulse action is positive on x only, with no x-axis feedback correction;
- y/z are weakly held to keep contact height and lateral alignment;
- per-step action deltas are limited.

This makes the push a single forward pulse instead of a high-frequency position-servo chase.

### Rollout-Target Dataset

For larger dataset construction, use a rollout-first target definition:

1. Run the robot with a sampled friction, push direction, push distance, and
   force setting.
2. Let the object stop.
3. Filter the rollout by reachable workspace, displacement range, and final
   object speed.
4. Use the final object xy as the target center.
5. Emit two cases with identical physics and action settings:
   - `observation_*`: target region is transparent, so the robot only observes
     the physical outcome.
   - `task_*`: target region is green and centered at the rollout endpoint.

Push direction is represented by `pusher_push_angle_deg`. The controller treats
the push offsets as local coordinates:

- `0 deg` is a straight push along table +x;
- angled pushes can be sampled in the `[-30 deg, 30 deg]` range;
- approach, contact, push, and retreat targets all rotate with the same local
  frame, so angled pushes do not require a separate controller.

Force diversity is produced by varying:

- `pusher_push_distance_x`;
- `pusher_push_steps`;
- `pusher_push_controller_scale`;
- friction coefficient;
- push angle.

For position pushes, smaller `pusher_push_steps` means the target advances more
per environment step, which makes the same path a stronger push. Larger
`pusher_push_steps` makes the push gentler.
`pusher_push_controller_scale` is kept as an explicit sweep dimension because
high-friction cases need a stronger stable controller, while low-friction cases
should use gentler pushes.

### Acceptance Checks

A case is accepted only if all checks pass:

1. `success = true` in `metadata.json`.
2. final distance is below `target_radius`.
3. push x action is never negative.
4. end-effector x does not move backward during the push phase.
5. visual video shows a single smooth push, not repeated high-frequency poking.
6. retreat and settle do not show IK / joint-angle jumping.

If a case fails because it cannot reach the target, tune `pusher_push_controller_scale`, `pusher_push_steps`, and `pusher_push_action_end` first. Do not solve it by enabling large global `controller_output_scale`.

## 2. Center-of-Mass-Randomized Push Dataset

This section is the design plan for the next dataset family. Unlike the first
chapter, the table friction is fixed and the object dynamics change through the
object center of mass (COM). The initial recommended table friction is:

```text
friction_mu = 0.06
```

The task should force the model to infer object-specific dynamics from visual
appearance and short interaction history. Therefore the object should not be a
plain symmetric box. Each object instance should have:

- a non-symmetric or at least orientation-readable footprint;
- visible surface texture, color panels, labels, or fiducial-like markings so
  the policy can distinguish object sides and contact parts from camera frames;
- a randomized but physically plausible COM offset in the object local frame;
- a fixed friction coefficient, so sliding / rotation differences are caused by
  COM and contact geometry rather than by table friction.

### Dataset Goal

For each sampled object and COM offset, collect pushes from multiple directions,
with multiple contact regions and push strengths. The important observed
outcomes are:

- final object xy displacement;
- final yaw change;
- full pose trajectory during and after contact;
- whether the object rotates around the heavy side, slides nearly rigidly, or
  tips / leaves the useful workspace.

As in the friction dataset, use a rollout-first target definition:

1. Sample object family, texture/appearance variant, COM offset, initial pose,
   push direction, contact region, push distance, push steps, and controller
   scale.
2. Run a hidden-target observation rollout at `friction_mu = 0.06`.
3. Let the object settle.
4. Filter by workspace bounds, displacement range, yaw range, final speed, and
   visual stability.
5. Use the final object xy, and optionally final yaw, as the paired task target.
6. Emit paired cases with identical physics and action settings:
   - `observation_*`: no visible target, used for dynamics observation.
   - `task_*`: visible green xy target, and later optionally a yaw/outline
     target if pose matching becomes part of the objective.

The first implementation can keep the target as xy-only to stay compatible with
the existing push-box collector, but the metadata must record final yaw and yaw
delta from day one.

### Object And COM Construction

There are two practical object-generation paths.

Recommended path for the first version:

- create generated object XMLs under a new asset directory such as
  `generated_assets/com_push_objects`;
- represent each object as a single free-joint body with textured visual geoms
  and simple collision geoms;
- use either an explicit `<inertial pos="...">` offset or an invisible dense
  ballast geom with `contype="0"` / `conaffinity="0"` to move the COM;
- keep total mass in a narrow range so the dominant variation is COM, not mass;
- copy or procedurally generate textures with clear front/back/left/right cues.

Fallback path:

- reuse stable LIBERO textured objects such as `cream_cheese`, `butter`,
  `macaroni_and_cheese`, `milk`, `ketchup`, and sauce cartons;
- generate patched XML copies with changed inertial / ballast settings;
- reject objects that tip too easily or have unreliable collision geometry.

The generated object catalog should record:

| field | meaning |
| --- | --- |
| `object_family` | shape class, e.g. box, carton, bottle, wedge, L-shape |
| `object_asset_id` | exact XML / texture variant |
| `footprint_half_extents_m` | approximate local x/y support dimensions |
| `mass_kg` | total object mass |
| `com_offset_local_m` | local-frame COM offset from geometric center |
| `com_offset_norm` | COM offset divided by local half extents |
| `com_bin` | `center`, `edge_x`, `edge_y`, or `corner` |
| `texture_id` | surface pattern / label variant |

Initial COM sampling ranges should stay inside the support polygon:

| bin | normalized local COM range | target share |
| --- | --- | ---: |
| `center` | `abs(x) <= 0.10`, `abs(y) <= 0.10` | 20% |
| `edge_x` | `0.20 <= abs(x) <= 0.40`, `abs(y) <= 0.20` | 25% |
| `edge_y` | `abs(x) <= 0.20`, `0.20 <= abs(y) <= 0.40` | 25% |
| `corner` | `0.20 <= abs(x) <= 0.40`, `0.20 <= abs(y) <= 0.40` | 30% |

These are starting ranges. The calibration pass should shrink them per object
family if a shape tips, rolls unpredictably, or produces too many rejected
rollouts.

### Push Sampling

The current push-box controller already supports push direction through
`pusher_push_angle_deg`, push strength through `pusher_push_distance_x`,
`pusher_push_steps`, and `pusher_push_controller_scale`. The COM dataset also
needs contact-region sampling, because pushing near or far from the heavy side
is the main signal.

New push parameters to add:

| field | meaning |
| --- | --- |
| `push_direction_deg` | world/table push direction |
| `contact_side` | side of the object approached by the gripper |
| `contact_lateral_fraction` | contact point along that side, from `-1` to `1` |
| `contact_region_bin` | `left`, `center`, or `right` along the contacted side |
| `push_distance_x` | commanded local push stroke |
| `push_steps` | number of push steps; fewer steps means stronger push |
| `push_scale` | push-phase controller scale |
| `force_bin` | `gentle`, `medium`, or `strong` |

Suggested first sampling grid:

| axis | values |
| --- | --- |
| friction | fixed `0.06` |
| push directions | `-135`, `-90`, `-45`, `0`, `45`, `90`, `135`, `180` deg, filtered by reachability |
| contact lateral fraction | `-0.6`, `0.0`, `0.6` |
| push distance | `0.08`, `0.12`, `0.16`, `0.20` m |
| push steps | `8`, `12`, `16`, `24`, `32` |
| push scale | start with `4`, `8`, `12`, then calibrate |

For shaped objects, `contact_side` and `contact_lateral_fraction` should be
converted into a local contact point using the object footprint. Approach,
contact, push, and retreat targets then rotate with the selected object yaw and
push direction.

### Calibration Table

Before collecting LeRobot episodes, create a no-video calibration table. It
should play the same role as
`configs/libero_push_box_push_calibration_table.md`, but with COM and contact
metadata added.

Proposed table columns:

| column | meaning |
| --- | --- |
| `object_family` | shape class |
| `asset_id` | exact generated object |
| `com_bin` | center / edge / corner class |
| `com_x_norm`, `com_y_norm` | normalized COM location |
| `direction_bin` | push direction bucket |
| `contact_bin` | contact location bucket |
| `force_bin` | gentle / medium / strong |
| `push_distance_m` | selected stroke |
| `push_steps` | selected step count |
| `push_scale` | selected controller scale |
| `actual_disp_cm` | final xy displacement |
| `actual_yaw_deg` | final yaw change |
| `valid` | accepted by filters |
| `reject_reason` | if invalid |

The final collector should sample from accepted rows rather than blind grid
search. Target outcome bins should be balanced approximately as:

| outcome bin | range | target share |
| --- | --- | ---: |
| short displacement | `8-18 cm` | 30% |
| mid displacement | `18-35 cm` | 40% |
| long displacement | `35-55 cm` | 30% |
| low yaw | `< 10 deg` | keep, but cap if overrepresented |
| medium yaw | `10-45 deg` | preferred |
| high yaw | `45-120 deg` | keep if stable and in workspace |

### Pilot Dataset

The first pilot should be small enough to inspect manually:

- 4 to 6 object families;
- 5 COM bins / offsets per family, including one centered COM baseline;
- 8 reachable push-direction bins;
- 3 contact bins per contacted side;
- 3 force bins;
- target roughly 100 accepted rollout pairs and 200 LeRobot episodes.

Balance quotas should be filled over:

```text
(object_family, com_bin, direction_bin, contact_region_bin, force_bin)
```

If this is too sparse, use the coarser key:

```text
(object_family, com_bin, direction_bin, force_bin)
```

and report contact-region counts separately.

### Acceptance Checks

A COM-push case is accepted only if all checks pass:

1. The object remains on the table and inside workspace bounds.
2. The final xy displacement is inside the selected useful range.
3. Final xy speed is below the stop threshold.
4. The object does not visually tip, jump, tunnel, or collide with the gripper
   after retreat.
5. The push action has no backward component along the selected push direction.
6. The end effector does not move backward during the push phase.
7. The final yaw is finite and recorded, even if the visible target is xy-only.
8. The object texture makes the contacted side and object orientation visible
   from the recorded camera views.

Do not accept a case simply because it reaches a target. The goal is coverage
over COM-dependent sliding and rotation behavior under fixed table friction.
