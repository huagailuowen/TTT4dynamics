# Dataset Generation Notes

## LIBERO Push-Box Friction Dataset

This dataset is a physical-property adaptation benchmark. The robot pushes a box on a table with different MuJoCo friction coefficients. The intended behavior is:

1. close the gripper;
2. move to the side of the box;
3. make one short, fast, smooth push;
4. retreat;
5. let the box slide freely into the green target region.

The push must not be generated by repeatedly moving the end effector backward and forward at high frequency. That behavior can reach the target, but it is hard for a policy to learn and does not represent a clean physical impulse. The current generator therefore uses a one-shot impulse push.

## Files

- `ttt4dynamics/push_box_libero.py`
  - LIBERO / MuJoCo environment wrapper.
  - Applies friction to the box and table collision geoms.
  - Implements scripted approach, descend, push, retreat, and settle phases.
- `scripts/sweep_libero_push_box_impulse.py`
  - Sweeps impulse-push parameters for one friction value.
  - Reports final distance, minimum distance, end-effector step size, and backward-motion checks.
  - Stops when it finds a successful case with no backward push oscillation.
- `scripts/generate_libero_push_box_adaptation_dataset.py`
  - Generates BDDL files and calibrated case configs for source/adaptation groups.
- `scripts/render_libero_push_box_friction_cases.py`
  - Renders cases to videos and writes `metadata.json`.

## Control Design

The old approach used large global `controller_output_scale` values. That is unsafe for this dataset: it changes the controller scale for all phases, including retreat and settle, and can make the IK / OSC solver jump between joint configurations.

The fixed approach separates high-speed pushing from normal arm motion:

- `controller_output_scale = 1.0`
- `enable_controller_output_scaling = false`
- normal approach / descend / retreat use the standard controller scale;
- only the push phase uses `pusher_push_controller_scale`;
- push uses `pusher_push_mode = "impulse"`;
- the impulse action is positive on x only, with no x-axis feedback correction;
- y/z are weakly held to keep contact height and lateral alignment;
- per-step action deltas are limited.

This makes the push a single forward pulse instead of a high-frequency position-servo chase.

## High-Friction Selected Cases

The current high-friction reference cases use:

- push stroke: `0.14 m`;
- target radius: `0.025 m`;
- initial xy: `(-0.245, -0.035)`;
- target xy: `(0.020, -0.035)`;
- no global controller scaling.

Selected cases:

| friction_mu | push_steps | push_scale | action_end | final distance | note |
| --- | ---: | ---: | ---: | ---: | --- |
| 0.20 | 4 | 12 | 0.45 | 0.92 cm | hardest tested case |
| 0.15 | 3 | 14 | 0.45 | 1.38 cm | high friction |
| 0.10 | 3 | 16 | 0.35 | 2.22 cm | high friction |

All selected cases satisfy:

- final distance is inside the 2.5 cm target radius;
- push action has no negative x command;
- end-effector x does not step backward during push;
- the box is pushed by one forward impulse, then slides/stops.

## Commands

Use the FastWAM virtualenv because it has LIBERO, robosuite, imageio, and the rendering dependencies installed.

```bash
export REPO=/inspire/hdd/project/robot-reasoning/xuyue-p-xuyue/cy/tool_adaptation_Project/TTTdynamics/repos
export PYTHONPATH=$REPO/LIBERO:$REPO/TTT4dynamics
export MUJOCO_GL=egl
export PY=$REPO/FastWAM/.venv/bin/python
cd $REPO/TTT4dynamics
```

Sweep the hardest friction case first:

```bash
$PY scripts/sweep_libero_push_box_impulse.py \
  --output configs/libero_push_box_mu2000_impulse_sweep.json \
  --friction 0.20 \
  --push-distance-x 0.14 \
  --push-steps 3 4 5 6 8 \
  --push-scales 4 6 8 10 12 14 16 18 20 \
  --action-ends 0.35 0.45 0.5 0.6 0.8 \
  --camera-resolution 24 \
  --max-steps 220
```

Render selected cases:

```bash
$PY scripts/render_libero_push_box_friction_cases.py \
  --cases configs/libero_push_box_high_friction_impulse_selected.json \
  --output-dir outputs/libero_push_box_high_friction_impulse_selected_compare \
  --camera both \
  --comparison-cols 3 \
  --fps 20
```

## Acceptance Checks

A case is accepted only if all checks pass:

1. `success = true` in `metadata.json`.
2. final distance is below `target_radius`.
3. push x action is never negative.
4. end-effector x does not move backward during the push phase.
5. visual video shows a single smooth push, not repeated high-frequency poking.
6. retreat and settle do not show IK / joint-angle jumping.

If a case fails because it cannot reach the target, tune `pusher_push_controller_scale`, `pusher_push_steps`, and `pusher_push_action_end` first. Do not solve it by enabling large global `controller_output_scale`.
